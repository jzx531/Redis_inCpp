这是一个非常敏锐的观察。即使你**注释掉了**后面读取数据的代码，程序依然会在第二次请求时崩溃。

### 🔴 核心原因：TCP 是“流”协议，不是“包”协议

崩溃的根本原因不在于你**有没有处理**读到的数据，而在于你**有没有把 socket 接收缓冲区里的数据取干净**。

#### 1. 你的代码做了什么？
```cpp
int32_t n = read_full(fd, rbuf, 4); // 只读了 4 字节（长度头）
// ... 解析 len ...
// 后面的 read_full 被注释掉了！
return 0; // 函数结束，回到主循环
```
**后果**：你从操作系统内核的接收缓冲区里只拿走了 **4 个字节**。剩下的 `len` 个字节（状态码 + 数据）**依然留在操作系统的接收队列里**，等待着下一次 `read`。

#### 2. 崩溃现场重现（第二次请求时）

假设协议格式：`[4字节长度] [4字节状态码] [数据]`
客户端发送两次：
1.  **包 1**: `00 00 00 0A` (10) + `00 00 00 00` (OK) + `1234` (4字节) -> 总共 14 字节。
2.  **包 2**: `00 00 00 0A` (10) + ...

**第一轮调用 `read_res` (处理包 1)**:
*   `read_full(fd, rbuf, 4)` 读走了前 4 字节：`00 00 00 0A`。
*   解析出 `len = 10`。
*   **关键点**：此时 OS 内核缓冲区里还剩下 **10 字节** (`00 00 00 00 12 34`)。
*   因为你注释掉了后面的读取，这 10 字节**原封不动地留在那里**。
*   函数返回 0，主循环继续。

**第二轮调用 `read_res` (处理包 2)**:
*   客户端发送了包 2。
*   此时 OS 内核缓冲区里的数据排列如下：
    `[旧包残留: 00 00 00 00 12 34]` + `[新包头部: 00 00 00 0A ...]`
*   代码再次执行：`read_full(fd, rbuf, 4)`。
*   **灾难发生**：`read` 函数不管什么“包边界”，它只管从缓冲区头部拿 4 个字节。
    *   它拿到的是：**`00 00 00 00`** (这是上一包的状态码！)。
*   **解析错误**：
    *   `memcpy(&len, &rbuf[0], 4)` -> `len` 变成了 **0**。
    *   如果运气不好，上一包残留的数据不是 0，而是随机值（比如 `00 00 10 00` = 4096）。
    *   `len` 变成了 **4096**。

**为什么会导致崩溃？**

这里有两种崩溃路径：

**路径 A：逻辑错乱导致的后续崩溃**
*   如果解析出的 `len` 是一个巨大的随机数（因为把数据当成了长度）。
*   虽然你注释掉了当前的 `read`，但你的主循环或者其他逻辑可能会基于这个错误的 `len` 做判断。
*   更常见的是，**错位会累积**。
    *   第 2 次读错了 4 字节。
    *   第 3 次读的时候，偏移量完全对不上。
    *    eventually，你会读到一个极大的 `len`，触发 `if (len > k_max_msg)` 检查失败返回 -1，或者如果检查有漏洞，后续的 `malloc(len)` 或数组访问直接 **Segmentation Fault**。

**路径 B：最直接的崩溃原因（通常发生在稍微修改代码后）**
哪怕你现在只是打印一下，看起来没崩。但只要你的程序逻辑依赖于“每次 `read_res` 处理一个完整包”，这种**错位**就是定时炸弹。
*   如果 `len` 解析为 0：代码可能跳过某些逻辑，导致状态机不同步。
*   如果 `len` 解析为极大值：
    *   假如你后来取消了注释，或者在其他地方用了 `len` 做缓冲区大小。
    *   `char big_buf[len]` -> 栈溢出。
    *   `read(fd, buf, len)` -> 试图读取几 GB 数据，填满内存或报错。

**但在你提供的这段特定代码中，为什么一定会崩？**
仔细看你的代码逻辑：
```cpp
    if (len < 4) {
        msg("Bad response: too short");
        return -1;
    }
```
如果在第二轮，读到的残留数据解析出来 `len < 4`（比如读到 `00 00 00 00`，len=0），函数返回 -1。
主循环收到 -1 通常会做什么？
```cpp
if (read_res(fd) < 0) {
    close(fd); // 关闭连接
    return;
}
```
**这不算“崩溃”（Crash/Segfault），这叫“连接意外断开”**。
如果你看到的現象是程序直接 **Segmentation Fault (核心转储)**，那么通常是因为：
1.  **len 解析成了一个大数**，且你在代码的其他地方（未展示部分）使用了这个 `len` 进行内存操作。
2.  或者，**多次错位后**，某次读到的数据恰好让 `len` 通过了检查，但随后的逻辑（比如主循环里根据 `read_res` 的返回值做的其他操作）访问了非法内存。

### 💡 形象的比喻

想象 TCP 数据流是一条**传送带**，上面放着一个个盒子（数据包）。
每个盒子前面贴着一个标签（4字节），写着盒子的长度。

*   **正确的做法**：
    1.  看标签（读 4 字节）。
    2.  根据标签长度，把整个盒子搬走（读 `len` 字节）。
    3.  传送带上干净了，下一个盒子正好在起始位置。

*   **你的做法（只读标签）**：
    1.  看标签（读 4 字节）：发现长度是 10。
    2.  **转身走了**，没搬盒子。
    3.  盒子的主体（10 字节）还留在传送带起始位置。
    4.  **下一次再来**：
        *   你又去看“标签”。
        *   但你看到的其实是**上一个盒子的主体部分**（比如盒子里的填充物）。
        *   你把填充物当成了长度标签。
        *   如果填充物是 `00 00 00 00`，你以为长度为 0。
        *   如果填充物是 `FF FF FF FF`，你以为长度为 42 亿！
        *   接下来任何基于这个“42 亿”长度的操作（比如申请内存、读取数据）都会瞬间炸毁程序。

### ✅ 解决方案

必须把数据读干净，让文件描述符的读取指针（Offset）移动到下一个包的开头。

取消注释并恢复逻辑：

```cpp
    // ... 前面的代码不变 ...
    
    // 【必须执行】读取剩余的 len 字节，将 socket 缓冲区清空到下一个包的边界
    n = read_full(fd, rbuf + 4, len); 
    if (n < 0) {
        return -1;
    }

    // 现在 rbuf 里才是完整的一个包： [4字节len][4字节code][数据]
    // 可以安全解析了
    uint32_t rescode = 0;
    memcpy(&rescode, &rbuf + 4, 4);
    
    // ... 打印逻辑 ...
```

### 总结
**不读完 = 留残数据 = 下次读错位 = 长度解析错误 = 内存越界/逻辑崩溃。**
TCP 不会帮你自动切分数据包，**你必须通过精确的 `read` 次数和字节数来维护包的边界**。少读一个字节，后续所有通信都会全部错位，直到程序崩溃。
